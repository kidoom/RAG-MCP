## 已学知识点详解

### D1.1 端到端数据流（评分 8.5 / 10）

**要能口述的链路**

- **入口分工**：根目录 `main.py` 偏启动提示与配置校验；离线建库走 `scripts/ingest.py`；命令行检索走 `scripts/query.py`；MCP 暴露工具走 `python -m mcp_server`（见 README）。
- **查询侧**：Hybrid（稠密 + 稀疏）→ **RRF 融合** → 可选 **Rerank**；CLI 可用 `--no-rerank` 跳过精排。
- **摄取侧**：需同时维护适合混合检索的索引（向量 + BM25）；`--force` 与完整性校验影响是否跳过已处理文件。
- **口述加分项**：报出 intake **阶段名**（integrity → load → split → transform → encode → store）及对应源码入口 `src/ingestion/pipeline.py`。

**会话留下的改进点**

- 面试时可主动点缀 **文件路径**，避免「只会概念、不落仓库」。

### D1.2 三层架构设计（评分 9 / 10）

**核心分层**

- **`core/`**：编排与领域契约（如 `types`、`query_engine`、`trace`）。检索 **策略**（例如稠密召回如何与稀疏结果合并）放在 core，而不是单一外部适配。
- **`libs/`**：厂商与 I/O **适配**（LLM、Embedding、VectorStore、Loader、Reranker 等）；**不应依赖 `ingestion`**，避免适配层反向耦合摄取编排。
- **`ingestion/`**：摄取流水线编排。

**DenseRetriever 为什么在 core**

- 表达的是 **检索编排与策略**（如何嵌入查询、如何与稀疏分支衔接），具体连哪家 API / 哪个向量库仍通过 `libs` + Factory 注入。

**表述时注意**

- DenseRetriever 仍会触及外部能力，但应强调 **经抽象接口封装**，而不是脚本里直接散落 SDK 调用。

### D1.3 Pipeline 组装（评分 9 / 10）

**Settings 与 YAML**

- `load_settings()` 默认读取仓库内 **`config/settings.yaml`**（路径相对仓库根、与当前工作目录无关），见 `src/core/settings.py` 中 `DEFAULT_SETTINGS_PATH`。
- `Settings.ingestion` 为 **Optional**：YAML **允许省略** `ingestion` 段时，`from_dict` 得到 `ingestion=None`。

**为何 ingestion 在 Settings 层可选**

- **全局 Settings** 会被多种入口复用；**纯查询 / MCP** 部署若强制写满 `chunk_size`、`splitter`、`batch_size` 等，会造成 **死配置与认知负担**。
- **摄取路径再收紧**：`IngestionPipeline` 初始化时若 `settings.ingestion is None` 则 **`ValueError`**，把「必须有摄取配置」推迟到 **真正运行摄取** 的边界。

**组装风格对比**

| 路径 | 组装位置 | 说明 |
|------|----------|------|
| 摄取 | `IngestionPipeline` 内部 | 默认注入 Chunker、Transform、BatchProcessor、存储等，CLI 侧 **`load_settings` → `IngestionPipeline(settings)`** |
| 查询 | `scripts/query.py` 的 `main()` | **显式**创建 `QueryProcessor`、`DenseRetriever`、`SparseRetriever`、`Fusion`、`HybridSearch`、`Reranker` |
| 根 `main.py` | — | **不组装**完整 RAG 流水线，仅加载配置与打印引导 |

**曾犯错误（已纠正）**

- 勿说「CLI 用不到 ingestion」——**离线摄取 CLI** 在调用 `IngestionPipeline` 时 **必须**有有效 `ingestion` 配置；可选的是 **YAML 是否声明**，不是摄取场景不需要。

### D1.4 核心数据类型（评分 9.5 / 10）

**四层类型总览**

| 类型 | 来源 | 关键增量字段 | 角色 |
|---|---|---|---|
| `Document` | Loader | `id` / `text` / `metadata` | 解析后的完整源文档 |
| `Chunk` | Splitter | `start_offset` / `end_offset` / `source_ref` | 带坐标的文本切片 |
| `ChunkRecord` | Embedder | `dense_vector` / `sparse_vector` | 带向量的入库单元 |
| `RetrievalResult` | QueryEngine | `chunk_id`（非 `id`）/ `score` | 查询命中结果 |

**标识字段语义**

- **`Chunk.id` / `ChunkRecord.id`**：块在摄取与存储侧的 **主键**（「我就是这个块」）。
- **`RetrievalResult.chunk_id`**：检索命中时对块的 **引用**（「我命中了这个块」），并带 **`score: float`**。

**RetrievalResult 为何更「瘦」**

- 不含 `Chunk` 的 `start_offset` / `end_offset` / `source_ref`。
- 不含 `ChunkRecord` 的 `dense_vector` / `sparse_vector`。  
→ 查询结果载荷只保留：**命中谁、多少分、给用户看的文本与 metadata**。

#### Document → Chunk：切分策略细节

**唯一实现**：`RecursiveCharacterSplitter`（`src/libs/splitter/recursive_splitter.py`）

**分隔符优先级链**（逐级降级）：

```
\n\n  →  \n  →  " "  →  ""（逐字符）
段落级     行级    词级     字符级
```

**递归切分算法**：
1. 尝试用 `\n\n` 切分文本
2. 若某片段仍超 `chunk_size`，用下一级分隔符继续切
3. 逐级降级，直到 `""`（纯字符滑窗兜底）
4. 切完后**分隔符追加回前一个 chunk**（保留上下文，最后一段除外）
5. `_apply_chunk_limits()` 做**小块合并** + **重叠注入**（`overlap_start = max(0, len(chunk) - chunk_overlap)`）

**默认参数**（`config/settings.yaml`）：`chunk_size=1000`、`chunk_overlap=200`、`splitter="recursive"`

**DocumentChunker 适配层**（`src/ingestion/chunking/document_chunker.py`）：
- 稳定 ID：`{doc_id}_{index:04d}_{md5(text)[:8]}`（同样文本同样 ID → 幂等入库）
- 元数据继承：父 Document 的 metadata 整体拷贝到每个 Chunk
- 图片引用分发：按 `[IMAGE: {id}]` 占位符把属于该 chunk 的 images 子集挂上去

**ChunkRefiner 后处理**（`src/ingestion/transform/chunk_refiner.py`）：
- 规则清洗（始终）：保留围栏代码块、去 HTML 注释/页码/分隔线、合并空白
- LLM 精修（可选 `use_llm: true`）：提升清晰度 + 补上下文；失败静默回退规则结果

---

#### Chunk → ChunkRecord：向量化细节

**BatchProcessor**（`src/ingestion/embedding/batch_processor.py`）按 `batch_size=100` 分批，每批**双路并行**：

| 路径 | 组件 | 做什么 | 产出 |
|---|---|---|---|
| Dense | `DenseEncoder` | 调 Embedding provider（默认 Qwen `text-embedding-v3`，1024 维） | `dense_vector: List[float]` |
| Sparse | `SparseEncoder` | 正则 `[A-Za-z0-9]+` 分词 → 小写 → 去停用词 → 统计词频 | `sparse_vector: {terms, term_weights, doc_length, unique_terms}` |

两路结果按 ID 匹配合并为单个 `ChunkRecord`，同时携带 `dense_vector` 和 `sparse_vector`。

**Embedding provider 可插拔**（`src/libs/embedding/embedding_factory.py`）：openai / azure / ollama / qwen / gemini / huggingface(bge) 六种，通过 `BaseEmbedding` + `EmbeddingFactory` 切换。

---

#### 类型注解 vs `validate_*_contract`

- **类型注解**：服务静态分析与 IDE（如 mypy），**不拦截运行时脏数据**。
- **`validate_document_contract` / `validate_chunk_contract`**：运行时契约，防御 Loader 产物、`from_dict` 反序列化等 **违反业务形状** 的输入（如缺 `source_path`、非法 `images` 列表）。两层各管各的。
- `validate_images_metadata`：参数为 `None` 时 **直接 return**（可选字段）。

**配套实现提醒**

- 图片占位：`IMAGE_PLACEHOLDER_TEMPLATE`、`extract_image_ids_from_text`。

**关联会话**：LEARNING_PROGRESS Detailed History #4，2026-05-15

### D2.1 IngestionPipeline 6 阶段（评分 9 / 10）

**管道流程**（`src/ingestion/pipeline.py`）

```
integrity → load → split → transform → encode → store
```

| 阶段 | 方法 | 做什么 | 关键组件 |
|---|---|---|---|
| 1. integrity | `_stage_integrity` | SHA256 去重检查，`--force` 可跳过 | `SQLiteIntegrityChecker` |
| 2. load | `_stage_load` | PDF → 文本 + 图片元数据 → `Document` | `PdfLoader`（在 libs） |
| 3. split | `_stage_split` | 按策略切割 Document → `Chunk[]` | `DocumentChunker` |
| 4. transform | `_stage_transform` | 三个子步骤顺序执行（见下） | ChunkRefiner → MetadataEnricher → ImageCaptioner |
| 5. encode | `_stage_encode` | Chunk → Dense 向量 + Sparse 向量 → `ChunkRecord[]` | `BatchProcessor` |
| 6. store | `_stage_store` | 写入向量库 + 构建 BM25 索引 | `VectorUpserter` + `BM25Indexer` |

**Transform 子步骤**：① `ChunkRefiner` 清洗去噪 → ② `MetadataEnricher` 补元数据 → ③ `ImageCaptioner` 生成图片文字描述。

**组装风格**：Ingest 把 7 个组件串在 `IngestionPipeline.__init__` 内部（黑盒），Query 把 6 个组件摊在 CLI 脚本里手写接线（白盒）。

## 检索与精排

### RRF 粗排（`src/core/query_engine/fusion.py`）

纯算法，无模型调用：`score(chunk) = Σ 1 / (rrf_k + rank)`，dense 和 sparse 两路各自的排名位置累加得分，按总分降序排列。同一 chunk_id 去重合并。

### Reranker 精排（`src/core/query_engine/reranker.py`）

三种后端（`src/libs/reranker/reranker_factory.py`）：

| 后端 | 实现 | 说明 |
|---|---|---|
| `llm` | `LLMReranker` | 把所有候选文档打包进 prompt，让 LLM 一次性排出 `{"ranked_ids": [...]}` |
| `cross_encoder` | `CrossEncoderReranker` | 用 `sentence-transformers` 的 CrossEncoder 逐对 (query, passage) 打分 |
| `none` | `NoneReranker` | 跳过精排，直接用 RRF 结果 |

**LLM 精排依据**（`config/prompts/rerank.txt`）：5 个维度 — 语义相似度、信息完整性、权威可信度、时效性、用户意图。但模型实际只拿到 `id + text[:4000]`，后三个维度全靠语义推断。

**容错**：无论选哪个后端，调用失败时自动 fallback 回 RRF 融合顺序。

**项目中未使用 MMR**：当前所有精排方案都贪 relevance，没有多样性去重逻辑。同一文档的多个相邻 chunk 排在 Top-K 里不会主动去重。

## 混合检索的工程取舍

### 为什么同时维护向量存储 + BM25 索引

> 语义召回靠向量，关键词精确匹配靠 BM25 — 两者互补，缺一个都会在特定 query 类型上丢精度。

| 维度 | Dense 向量 | BM25 稀疏 |
|---|---|---|
| 强项 | "怎么配环境" → 语义匹配"配置指南" | "Error 0x8007" → 精确命中日志原文 |
| 弱项 | 专业术语、错误码、版本号匹配差 | 同义词、改写、跨语言无能为力 |

Ingestion 阶段同时产出两路索引，Query 阶段用 RRF 融合两路结果，各取所长。

## MCP 服务协议

### 调用链路（`src/mcp_server/server.py`）

```
智能体 → stdin JSON-RPC → server.py run_stdio_server()
  → protocol_handler.dispatch(payload)
    → method="tools/list"     → 返回 3 个 ToolSpec schema
    → method="tools/call"     → 按 name 路由到 handler(query_knowledge_hub 等)
    → response → stdout
```

### 注册的 3 个 Tool（`src/mcp_server/tools/__init__.py`）

| Tool | 功能 |
|---|---|
| `query_knowledge_hub` | 检索知识库并返回带引用的答案 |
| `list_collections` | 列出可用文档集合 |
| `get_document_summary` | 根据 doc_id 获取文档摘要 |

### ToolSpec 如何让智能体"知道用法"（`src/mcp_server/protocol_handler.py`）

`ToolSpec` 包含 `name + description + inputSchema`（JSON Schema 格式），`tools/list` 时序列化返回。`handler` 字段只在服务端内部路由，不发给智能体。

## Response 组装层（`src/core/response/`）

3 个组件把 `RetrievalResult[]` 转为 MCP 结构响应：

| 组件 | 作用 |
|---|---|
| `CitationGenerator` | 提取 source、page、chunk_id、score → 结构化引用 |
| `MultimodalAssembler` | 从 chunk 元数据提取 image_refs → base64 编码 → 塞入 content |
| `ResponseBuilder` | 协调上述两者，输出 `{content, structuredContent}` |

## 评估体系（`scripts/evaluate.py` + `src/observability/evaluation/`）

### 黄金测试集（`tests/fixtures/golden_test_set.json`）

手工标注 (query, expected_chunk_ids) pair，跑检索后比对命中情况。当前数据是 Demo 占位符，chunk ID 与真实摄入不匹配。

### 评估指标（`src/libs/evaluator/custom_evaluator.py`）

| 指标 | 计算 | 含义 |
|---|---|---|
| Hit Rate | 返回列表里有没有命中黄金 chunk | ≥1 命中 = 1.0 |
| MRR | 第一个命中黄金 chunk 排名的倒数 `1/rank` | 排第 1 = 1.0，未命中 = 0.0 |

### 局限性

- 黄金测试集无通用性，换知识库需重新标注
- 只评估检索排名质量，不评估 LLM 生成答案质量

## 易错点索引（复习用）

| 标签 | 要点 |
|------|------|
| Optional ≠ 不用 | `ingestion` 省略仅表示全局配置可精简；跑摄取仍会失败若仍为 `None` |
| main vs scripts | `main.py` 不承担 ingest/query 编排 |
| core vs libs | 策略/编排 vs 适配；`libs` 不依赖 `ingestion` |
| 口述 | 关键结论尽量带 **路径 + 符号**（类名、阶段名、`--flag`） |
| RRF ≠ Rerank | RRF 是粗排纯算法，Reranker 是精排调模型，两者不可混为一谈 |
| 黄金测试集 | 当前是 Demo 数据，chunk ID 对不上真实 intake，评估结果无意义 |

```
  │   ↓           │    │   ↓             │
  │ 向量库+BM25   │    │ Response        │
  └───────────────┘    └─────────────────┘
```

> **读图口诀**：配置驱动一切（Settings）；写路径 6 阶段串行入库；读路径两路并行 → RRF 合并 → 可选 Rerank；MCP / Dashboard 是外部暴露面。core + libs 提供能力，ingestion / query_engine 做编排。

## 面试口述稿（约 90 秒）

> 配置统一由 **`Settings`** 驱动，`core/types.py` 是零依赖类型契约，`core/trace/` 做全链路追踪——这三块构成项目内部契约层。
>
> **`libs/`** 是外部适配层：`PdfLoader` 产出 `Document`，Embedding / VectorStore 等通过 **`Base*` + Factory** 可插拔切换。**libs 不依赖 ingestion**：适配层不知道编排层。
>
> **摄取管线**在 `src/ingestion/pipeline.py`，6 阶段——integrity → load → split → transform → encode → store——同时写入 **向量库与 BM25 索引**，支撑后续混合检索。
>
> **查询管线**在 `core/query_engine/`：`QueryProcessor` 做规范化，`DenseRetriever` + `SparseRetriever` 两路并行召回，**`Fusion` 用 RRF 融合排名**，`HybridSearch` 统一编排，可选 **`Reranker` 精排**（支持 LLM / CrossEncoder / none 三种后端）。
>
> **`DenseRetriever` 放在 core 而非 libs**：它表达的是检索策略与编排，具体连哪家厂商通过 libs + Factory 注入——与 `PdfLoader` 放在 libs（纯 I/O 适配）形成对称分工。
>
> **MCP Server** 在 `mcp_server/server.py`，stdio JSON-RPC，3 个工具（`query_knowledge_hub` / `list_collections` / `get_document_summary`）把读侧能力暴露给 IDE 和助手。`core/response/` 负责将检索结果组装为带引用和图片的结构化响应。

追问时可各补一句文件名：`ingestion/pipeline.py`、`query_engine/fusion.py`、`query_engine/dense_retriever.py`、`mcp_server/server.py`。
