架构 · 谁与谁对话

用户 / Agent / IDE（宿主）经 MCP Client 与本 Server 会话；服务端由工具处理器衔接核心检索管线。

数据流 · 载荷如何传递

从 MCP 解析出的参数一路变形为向量/BM25 候选，再汇合为最终结果列表与对外 JSON。

核心类型演变 · 查询侧

query_processor.ProcessedQuery → 各召回器产出 RetrievalResult · ResponseBuilder.build。

  • MCP arguments：`query` · `top_k`（可由工具裁剪上限）· 预留 filters 等与 ProcessedQuery.filters 对齐字段。
  • ProcessedQuery：raw_query / normalized_query / keywords / filters · 传给 Hybrid 与稀疏统计路径。
  • RetrievalResult：chunk_id · score · text · metadata · 多条组成精排输入与最终引用来源。

以磁盘上的 PDF为例：PdfLoader先把二进制页变成整篇文本 + 元数据契约（含[IMAGE: id]占位），再在进入向量库/BM25 前逐级变成 Chunk · ChunkRecord。

架构 · 摄取链路挂载点

Loader 向上对接文件系统；流水线向下对接嵌入与两处索引（dense + sparse）。

数据流 · PDF 如何变成可查片段

页序拼接 → Transform 可选增强 → 批编码 → Upsert。

数据结构演变 · PDF 示例

core.types.Document → Chunk → ChunkRecord；与 libs.splitter 内部 chunk 的差异在 ingest 中会映射到契约类型。

  • 磁盘：.pdf · PdfLoader.load(path) 读分页 reader。
  • Document（首次定型）：唯一 id；text=全文（页间用换行）；metadata 至少含 source_path · page_count；正文里可穿插 [IMAGE: …] 占位；可选 metadata.images[] · 每项记录 id · path · text_offset · text_length · 以及与页对齐的摘录信息。
  • Chunk：start_offset/end_offset 锁回原文；· source_ref 可追溯父文档；· metadata 随 Refiner · Enricher · Captioner 增补。
  • ChunkRecord：在文本与 meta 基础上附加 dense_vector 与稀疏侧统计向量 / 条目，再由 VectorUpserter 与 BM25Indexer 落盘。
