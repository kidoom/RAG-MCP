Project Study Guide

把 RAG、MCP、摄取、评估拆成可以逐层读懂的工程链路

这个项目不是单纯的 RAG Demo，而是一个模块化服务框架：PDF 等文档先被摄取成 Chunk、向量和 BM25 索引； 查询时走 Dense + Sparse 混合召回、RRF 融合、Rerank 精排；最后通过 MCP JSON-RPC 暴露给 Copilot、Claude 等客户端。

2Dense / Sparse 双检索通路
5默认最终返回 Top-K
60RRF 默认平滑参数
3MCP Tool 对外暴露

先看什么

学这个项目最容易卡住的地方，是把“模型能力”和“工程链路”混在一起。建议按下面顺序读：先抓主流程，再看每一步的输入输出，最后再替换 Provider 或调参。

1. 跑通配置理解 config/settings.yaml 如何决定 LLM、Embedding、VectorStore、Rerank。
2. 读摄取链路文件如何变成 Document、Chunk、ChunkRecord、Chroma 向量和 BM25 索引。
3. 读查询链路QueryProcessor、DenseRetriever、SparseRetriever、Fusion、Reranker 如何串起来。
4. 读 MCP 封装JSON-RPC 的 initialize、tools/list、tools/call 如何映射到 Python 函数。
5. 看观测与测试TraceContext、Dashboard、unit/integration/e2e 测试如何帮助定位问题。
学习主线：这套代码最值得学的是“接口抽象 + 配置驱动 + 降级保护 + 可观测性”。模型只是其中一个可替换部件。

查询链路

查询链路是这个项目最核心的学习对象。它的关键不是“调用一个向量库”，而是把多种召回方式统一成同一种结果格式，再用排序融合和精排提升质量。

1. QueryProcessor
将原始查询做空白规范化，提取英文/数字/中文 token，过滤英文停用词，并清理 metadata filters。 输出 ProcessedQuery(raw_query, normalized_query, keywords, filters)。
src/core/query_engine/query_processor.py
2. HybridSearch
用 ThreadPoolExecutor(max_workers=2) 并行启动 Dense 与 Sparse。实际召回量使用 recall_k = max(top_k * 2, top_k)，给后续融合留余量。
src/core/query_engine/hybrid_search.py
3a. DenseRetriever
用配置中的 Embedding Provider 把 query 编成向量，再查 VectorStore。返回统一的 RetrievalResult。
src/core/query_engine/dense_retriever.py
3b. SparseRetriever
查询 BM25 索引拿到 chunk_id，再从 VectorStore 批量取 text 和 metadata，保证 Sparse 结果也能进入统一排序链路。
src/core/query_engine/sparse_retriever.py
4. Fusion
对 Dense 和 Sparse 排名做 RRF：同一个 chunk 在两路都靠前时，分数会累加。排序 tie-breaker 使用 chunk_id，结果稳定可测。
src/core/query_engine/fusion.py
5. Reranker
默认按配置构建 LLM Reranker。失败时回退到候选原始顺序，并在 metadata 标记 fallback，避免整条查询链路崩掉。
src/core/query_engine/reranker.py
6. ResponseBuilder
生成 MCP content：文本 Markdown、citations 结构化字段，以及从 image_refs 拼装出的图片内容。
src/core/response/response_builder.py

查询链路伪代码

def query_knowledge_hub(query, top_k=5):
    settings = load_settings()
    processed = QueryProcessor().process(query)

    recall_k = max(top_k * 2, top_k)
    dense_results  = DenseRetriever(settings).retrieve(processed.normalized_query, recall_k)
    sparse_results = SparseRetriever(settings).retrieve(processed.keywords, recall_k)

    candidates = Fusion(settings).fuse(dense_results, sparse_results, top_k=recall_k)
    ranked = Reranker(settings).rerank(query, candidates, top_k=min(top_k, settings.rerank.top_k))

    return ResponseBuilder().build(ranked, query)
为什么 Dense 和 Sparse 都要保留

Dense 适合同义表达、语义相近的问题，比如“如何接入客户端”可以命中“Claude Desktop 配置”。Sparse 适合专有名词、函数名、配置键，比如 rrf_k、query_knowledge_hub、TraceContext。混合检索的收益来自互补，而不是简单叠加。

RRF 的实现细节

RRF 不直接比较 Dense score 和 BM25 score，因为两者量纲不同。它只关心排名：1 / (rrf_k + rank)。如果同一个 chunk 同时在 Dense 第 3 名、Sparse 第 2 名，它会得到两份分数。

aggregate_scores[chunk_id] += 1.0 / (self._rrf_k + rank)
ranked_ids = sorted(aggregate_scores, key=lambda id: (-aggregate_scores[id], id))

RAG 检索整体实现全流程

下面这节按真实调用顺序写。你可以把它当成源码阅读路线：每读完一步，就去右侧源码文件里看对应函数。

阶段 0：MCP Tool 入口 query_knowledge_hub(arguments) 读取 query 和 top_k，top_k 最大限制为 20，然后加载配置并组装各个组件。
阶段 1：查询预处理 QueryProcessor 负责把用户问题变成两类输入：Dense 用 normalized_query，Sparse 用 keywords。
阶段 2：双路粗召回 HybridSearch 用线程池并行跑 DenseRetriever 和 SparseRetriever，把两路结果统一成 RetrievalResult。
阶段 3：RRF 融合 Fusion 不比较原始分数，只比较排名，用 1/(rrf_k+rank) 把两路候选合并。
阶段 4：Rerank 精排 Reranker 默认使用 LLM 后端；失败时返回融合排序前几名，并标记 fallback。
阶段 5：响应构建 ResponseBuilder 输出 MCP content 和 structuredContent，包含 markdown、citation 和可选图片内容。

逐步展开

0. Tool 参数
输入来自 MCP tools/call：{"query": "...", "top_k": 5}。 代码会校验 query 非空，校验 top_k 为正数，并执行 top_k = min(top_k, 20)。
src/mcp_server/tools/query_knowledge_hub.py
1. 组件装配
每次调用都会 load_settings()，然后创建 QueryProcessor、DenseRetriever、SparseRetriever、Fusion、HybridSearch、Reranker。 Retriever 与 Reranker 内部再通过 Factory 创建具体 Provider。
src/core/settings.py
src/libs/*/*_factory.py
2. QueryProcessor
_normalize_query() 把连续空白折叠成一个空格；_extract_keywords() 用正则提取英文、数字、下划线和中文块； _normalize_filters() 清除空 key、None、空字符串。
src/core/query_engine/query_processor.py
3. Dense 召回
DenseRetriever 调用 embedding_client.embed_query(query) 得到 query_vector，再调用 vector_store.query(vector, top_k, filters)。返回结果会被包装成统一的 RetrievalResult。
src/core/query_engine/dense_retriever.py
4. Sparse 召回
SparseRetriever 用 keywords 查询 BM25，拿到 chunk_id 和 BM25 score；再通过 vector_store.get_by_ids(ids) 补齐 text 与 metadata，所以后面可以和 Dense 共用排序逻辑。
src/core/query_engine/sparse_retriever.py
src/ingestion/storage/bm25_indexer.py
5. 故障降级
HybridSearch 会分别捕获 Dense 和 Sparse 异常：只要其中一路成功，就继续走；两路都失败才抛错。 这就是检索层的 graceful degradation。
src/core/query_engine/hybrid_search.py
6. RRF 融合
Fusion 按排名遍历两路结果，用 chunk_id 聚合分数；同 id 的候选保留第一份 payload，但分数累加。 排序键是 (-score, chunk_id)，所以测试结果稳定。
src/core/query_engine/fusion.py
7. metadata filter
融合后再执行 metadata filter。普通值用相等比较，列表值支持交集匹配。最后返回 filtered[:top_k]。
HybridSearch._apply_metadata_filters()
8. Reranker
候选被转换成 {id, score, text, metadata} payload。LLM Reranker 读取 config/prompts/rerank.txt， 要求模型返回 {"ranked_ids": [...]}；如果失败，返回候选原顺序的前 k 个。
src/core/query_engine/reranker.py
src/libs/reranker/llm_reranker.py
9. Response
CitationGenerator 从 metadata 里提取 source、page、chunk_id；MultimodalAssembler 读取 image_refs 对应图片； ResponseBuilder 组合成 MCP 要求的 content 和 structuredContent。
src/core/response/

关键实现细节

为什么先粗召回再精排Embedding、BM25 成本低，适合扩大候选池；LLM Rerank 成本高，适合只处理少量候选。
为什么 Sparse 要回查 ChromaBM25 只负责倒排命中和打分，完整文本、metadata 仍由 VectorStore 统一提供。
为什么 RRF 不用原始 scoreDense 相似度和 BM25 分数不可直接比较，排名融合更稳。
为什么要稳定 chunk_idDense、Sparse、引用、图片、多轮摄取都靠同一个 chunk_id 对齐。
为什么要 trace检索质量差时，trace 能看出是关键词、向量库、BM25、RRF 还是 Rerank 的问题。
为什么 MCP 层要薄MCP 只做协议适配，核心逻辑留在 core/libs，方便 CLI、Dashboard、测试复用。

数据摄取

摄取链路决定“能不能搜到”。查询再聪明，如果 Chunk 质量、metadata、索引一致性做不好，召回和引用都会不稳定。

1. Integrity
计算文件 SHA256，SQLite 记录成功/失败状态；未变化时可跳过重复摄取。
libs/loader/file_integrity.py
2. Load
PDF Loader 读取文本、图片信息、页码和 source_path，形成 Document。
src/libs/loader/pdf_loader.py
3. Split
按配置的 splitter、chunk_size、chunk_overlap 切分，并生成稳定 chunk_id。
src/ingestion/chunking/document_chunker.py
4. Transform
依次执行 ChunkRefiner、MetadataEnricher、ImageCaptioner。可以用规则或 LLM 增强文本与 metadata。
src/ingestion/transform/
5. Encode
BatchProcessor 分批运行 DenseEncoder 和 SparseEncoder，再合并成同 id 的 ChunkRecord。
src/ingestion/embedding/batch_processor.py
6. Store
先 VectorUpserter 写 Chroma，拿到稳定 ids 后再构建 BM25，保证两套索引用同一 chunk_id 空间。
src/ingestion/storage/
关键工程点：IngestionPipeline._stage_store() 先 upsert 向量库，再把返回的稳定记录交给 BM25。这样 Sparse 命中的 chunk_id 才能回查到同一个 Chroma 记录。

MCP 服务

协议处理流程

  1. server.py 从 stdin 逐行读取 JSON-RPC 请求，stdout 只输出 JSON 响应，日志走 stderr。
  2. ProtocolHandler.dispatch() 根据 method 分发到 initialize、tools/list、tools/call。
  3. tools/__init__.py 注册三个 ToolSpec，并把 handler 绑定到具体 Python 函数。
  4. query_knowledge_hub 组装完整 RAG 链路并返回 MCP content。

配置入口

config/settings.yaml 是理解可插拔架构的入口。绝大多数替换都应该先从这里开始，而不是直接改业务代码。

配置块当前职责影响的源码
llm生成模型，也是 LLM Reranker 的后端来源。libs/llm/*、core/query_engine/reranker.py
embedding查询向量和文档向量的模型、维度、API 地址。libs/embedding/*、dense_retriever.py

retrievalDense/Sparse/Fusion Top-K 与 RRF k。hybrid_search.py、fusion.py

查询 trace 里最该看的字段

query_processing.keywords关键词抽取是否符合预期，Sparse 检索是否有输入。
dense_retrieval.result_count向量库是否真的返回结果，Embedding/Chroma 是否正常。
sparse_retrieval.errorBM25 索引是否加载失败，或者没有对应 chunk_id。
rerank.fallbackLLM Reranker 是否失败，是否退回到了融合原始顺序。

扩展点

这个项目的扩展姿势，是“新增实现 + 工厂注册 + 配置切换”。这样做可以把业务逻辑留在 core，把供应商差异放在 libs。

新增 Embedding Provider

  1. 在 src/libs/embedding/ 新增实现，遵守 BaseEmbedding 接口。
  2. 在 embedding_factory.py 注册 provider 名称。
  3. 在 settings.yaml 切换 embedding.provider、model、dimensions。
  4. 重新摄取数据，保证向量维度一致。

新增 MCP Tool

  1. 在 src/mcp_server/tools/ 新增 handler，输入输出都是 dict。
  2. 在 tools/__init__.py 添加 ToolSpec 和 inputSchema。
  3. 用 scripts/test_mcp_list.py 或集成测试验证 tools/list 与 tools/call。
